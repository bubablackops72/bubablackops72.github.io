Overview

What Is OllamaVoice?

OllamaVoice is a Windows desktop application that chains together three locally-running AI systems into a single voice interface: you speak, the app transcribes your voice using OpenAI Whisper, sends the text to a large language model running in Docker via Ollama, and reads the response back to you using Kokoro TTS in a deep British male voice — the closest local approximation of Jarvis from the Iron Man films.

Everything runs on your own hardware. No cloud calls, no API keys, no subscriptions. After the initial model downloads your internet connection is irrelevant.

Changelog

What Changed in v1.2

Version 1.2 focused entirely on getting Jarvis's voice actually working. What seemed like a simple TTS integration turned into a multi-session debugging effort involving Python version compatibility, PyInstaller frozen environments, Windows file permissions, and process scoping issues. Here's what was fixed and added.

Arch
Kokoro runs as a Python 3.11 subprocess, not bundled in the exe

PyInstaller bundles code into a frozen environment that breaks Kokoro's model download cache paths. The fix: the exe calls Python 3.11 directly as a subprocess for every TTS request. Kokoro runs in its full environment with all HuggingFace cache paths intact.

Fix
config.json moved to %APPDATA%\OllamaVoice\

The app was crashing at startup with Permission Denied when trying to write config.json to C:\Program Files\OllamaVoice\ — a read-only directory. Config and setup flags now live in AppData which is always writable.

Fix
Install path corrected to 64-bit Program Files

Inno Setup was defaulting to Program Files (x86), causing the app to run old cached versions. Added ArchitecturesInstallIn64BitMode and switched to {autopf64} to force the correct 64-bit install path.

Fix
tempfile scope collision between /speak and /transcribe handlers

The import tempfile inside the /speak try block was bleeding into the /transcribe handler's local scope, causing UnboundLocalError crashes. Fixed by aliasing to _tf inside the /speak block.

New
Speed control — 0.2x to 5.0x with 14 increments

A dropdown in the header lets you dial Jarvis's speaking pace from whisper-slow to rapid-fire. The selected speed is passed directly to Kokoro's speed parameter and persisted in localStorage between sessions.

New
Stop button — instantly silence Jarvis mid-sentence

A [|| STOP] button appears in the header while Jarvis is actively speaking. One click pauses the current audio and clears the entire speech queue. The button disappears automatically when speech ends.

New
Build cache auto-cleared on every compile run

create_installer.bat now deletes the build/ and dist/ folders before running PyInstaller, guaranteeing a fresh recompile every time. No more running old cached code after source changes.

Arch
Debug log at %TEMP%\ollamavoice_debug.log

All TTS events, subprocess calls, exit codes, and errors are written to a persistent log file regardless of console buffering. Essential for diagnosing issues in a PyInstaller exe where stdout can be unreliable.

Context

Why Voice Makes It Feel Real

"The gap between a chat interface and a genuine AI assistant isn't intelligence — it's presence. Voice closes that gap."

Text-based chat with a local LLM is impressive technically, but it still feels like using a search engine. You type, you wait, you read. The interaction is transactional. Adding voice changes the dynamic entirely — the model responds in a way that occupies the same sensory channel as human conversation.

The choice of Kokoro's bm_george voice isn't arbitrary. A deep, measured British male voice carries specific cultural associations: authority, composure, intelligence. These associations prime you to perceive the model's responses as more considered, more deliberate — even when the underlying model is identical to what you'd access through a text interface.

The speed control matters for the same reason. At 0.6x Jarvis sounds contemplative, like he's choosing words carefully. At 1.4x he sounds urgent, operational. The ability to tune this lets you match the assistant's voice to the nature of the task — a subtle but real contribution to perceived intelligence and "lifelikeness."

The Full Local Stack

What makes OllamaVoice different from connecting to GPT-4 with voice is the complete absence of external dependencies at runtime. The Whisper model runs on your CPU. The LLM runs on your GPU inside Docker. Kokoro renders audio on your Python 3.11 installation. Nothing leaves your machine. This matters for privacy, latency, and cost — but also for the nature of the experience. There's no rate limit, no server lag, no account. Just your hardware, talking to itself.

Roadmap

Where This Goes Next

The current architecture is a solid single-machine implementation. The bottlenecks are well understood and the paths to improvement are clear — though some require hardware investment that doesn't make financial sense yet.

01

GPU-Accelerated TTS

Kokoro currently runs on CPU via a Python subprocess. A second GPU — even a mid-range card — could render audio in real time instead of the current 2-10 second latency. With CUDA acceleration, Kokoro generates audio at roughly 50x real-time speed. A 200-word response would render in under a second.

02

Dedicated Audio GPU

The ideal split: primary GPU (RTX 5090) handles LLM inference exclusively, secondary GPU handles Whisper + Kokoro simultaneously. This eliminates all VRAM competition and allows the LLM to start streaming tokens while audio for the previous sentence is still rendering — true pipeline parallelism.

03

Multi-GPU Model Sharding

Ollama supports tensor parallelism across multiple GPUs. With two RTX 5090s (64GB combined VRAM), you could run 70B parameter models at full precision — a significant quality jump over the current 32B Q4 models. Response coherence and reasoning depth improve substantially above 30B parameters.

04

Parallel Specialist Models

Enough VRAM enables simultaneous model loading — a fast 7B model for quick factual queries, a 32B model for complex reasoning, a code-specialized model on standby. A routing layer selects the right model per query. The voice interface stays unified; the backend becomes a model farm.

The Hardware Equation

The honest constraint is economics. A second RTX 5090 at current pricing is ~$2,000-3,000. The marginal improvement over a single card — while real and measurable — doesn't justify that cost for a personal assistant project today. The architecture is ready for it, though. The Ollama Docker setup already passes --gpus all; adding a second card requires zero code changes.

The more interesting near-term opportunity is the dedicated audio GPU path. A used RTX 3060 or 3070 for $200-300 would completely eliminate the TTS latency bottleneck and make the voice interaction feel genuinely real-time. That's a near-term upgrade worth considering.

"The infrastructure for a truly capable home AI assistant exists today. The question is just how much of it you want to run at once — and what that's worth to you."

Technical Notes

Implementation Details

Why Python 3.11 for TTS

Kokoro requires PyTorch, which in turn requires specific CUDA and numpy versions. Python 3.14 (the system default on this machine) had no compatible prebuilt wheels for numpy 2.x at the time of development, requiring compilation from source — which fails without a C compiler. Python 3.11 has full prebuilt wheel support for the entire dependency chain. Rather than downgrade the system Python, we install 3.11 in parallel and invoke it as a subprocess specifically for TTS.

The Subprocess Architecture

Each call to /speak spawns a Python 3.11 process that imports Kokoro, generates audio, writes it to a temp WAV file, and exits. The main process reads the WAV and streams it to the browser. This is slower than keeping Kokoro resident in memory, but it completely avoids the PyInstaller frozen environment issues that prevented the in-process approach from working. A future version could run a persistent Kokoro server process to eliminate the per-call startup overhead.

GPU Configuration

The Ollama container is configured with five environment variables that together force 100% GPU utilization: OLLAMA_NUM_GPU=999, OLLAMA_GPU_LAYERS=999, OLLAMA_MAIN_GPU=0, OLLAMA_FLASH_ATTENTION=1, and OLLAMA_KV_CACHE_TYPE=q8_0. The last two are critical — flash attention and 8-bit KV cache quantization together reduce VRAM usage enough that all 32B model layers fit on a single 32GB card without spilling to CPU.
