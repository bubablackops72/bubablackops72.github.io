Fixing NemoClaw Onboarding on Windows 11 with WSL2 & Docker

How I got past step 5/7 of the NemoClaw setup — fighting through Linux capability errors, missing binaries, and Docker build failures that only show up in a WSL2 environment.

Setting up NemoClaw's OpenClaw agent on Windows 11 via WSL2 and Docker Desktop sounds straightforward — until it isn't. The official onboarding flow gets to step 5/7 and then hits a wall of Linux-specific errors that the docs don't cover. Here's every obstacle I ran into and exactly how I fixed each one.

The Obstacles & Solutions

Round 1
🚧 CAP_SETPCAP Wall
unable to raise CAP_SETPCAP for BSET changes: Operation not permitted

Root Cause: OpenClaw's security hardening tried to drop Linux capabilities including CAP_SETPCAP, which WSL2 blocks due to its virtualized kernel.

Fix: Edit ~/.nemoclaw/source/scripts/nemoclaw-start.sh and bypass the capability-dropping block:

nemoclaw-start.sh
# Changed from:
if [ "${NEMOCLAW_CAPS_DROPPED:-}" != "1" ] && command -v capsh >/dev/null 2>&1; then

# To:
if false && [ "${NEMOCLAW_CAPS_DROPPED:-}" != "1" ] && command -v capsh >/dev/null 2>&1; then

Result: Container startup progressed past the capability initialization phase.

Round 2
🚧 The Missing Binary
command -v openclaw returned empty (binary not found in PATH/runtime)

Root Cause: While openclaw existed during the Docker build phase, it wasn't installed globally or persisted into the final runtime image.

Fix: Updated the Dockerfile builder stage to explicitly install the CLI globally:

Dockerfile (builder stage)
WORKDIR /opt/nemoclaw
RUN npm install          # Build dependencies
RUN npm run build        # Compile TypeScript → dist/
RUN npm install -g openclaw  # Global CLI installation

Result: /usr/local/bin/openclaw became available at runtime.

Round 3
🚧 npm ci vs. package-lock.json
npm error: The npm ci command can only install with an existing package-lock.json

Root Cause: The build script used npm ci --omit=dev but only copied package.json — not package-lock.json — into the build context.

Fix (Option 1 — Preferred): Copy the lockfile in setup.sh:

setup.sh
cp "$REPO_DIR/nemoclaw/package.json" "$BUILD_CTX/"
cp "$REPO_DIR/nemoclaw/package-lock.json" "$BUILD_CTX/" 2>/dev/null || true

Fix (Option 2 — Quick): Swap npm ci --omit=dev → npm install --omit=dev in your Dockerfile. Less strict but gets the job done.

Result: Dependencies installed correctly without lockfile errors.

Round 4
🚧 Missing dist/ Directory
COPY failed: stat opt/nemoclaw/dist/: file does not exist

Root Cause: The Docker build copied source code but never ran the TypeScript compilation step, so /dist/ was empty when the runtime stage tried to copy from it.

Fix: Explicitly add the build step in the Dockerfile:

Dockerfile
RUN npm install
RUN npm run build   # Critical: generates /dist/
RUN npm install -g openclaw

Result: /opt/nemoclaw/dist/ populated correctly and the COPY succeeded.

Final Working Dockerfile

Here's the complete Dockerfile snippet that ties all four fixes together:

Dockerfile (full)
# Builder stage
FROM node:22-slim AS builder
WORKDIR /opt/nemoclaw
COPY nemoclaw/package.json nemoclaw/tsconfig.json ./
COPY nemoclaw/src/ ./src/
RUN npm install
RUN npm run build  # ← Non-negotiable for TS projects

# Runtime stage
FROM ghcr.io/nvidia/nemoclaw/sandbox-base:latest
WORKDIR /opt/nemoclaw
COPY --from=builder /opt/nemoclaw/dist/ ./dist/
COPY nemoclaw/openclaw.plugin.json ./
COPY nemoclaw/package.json nemoclaw/package-lock.json ./
RUN npm install --omit=dev
RUN npm install -g openclaw  # ← Ensures CLI availability
ENTRYPOINT ["/usr/local/bin/nemoclaw-start"]

Key Takeaways for WSL2 Users

  • WSL2 ≠ Bare Metal Linux. Capability manipulation like CAP_SETPCAP and seccomp profiling often fails — be ready to bypass hardening scripts.
  • Separate build vs. runtime stages clearly. Builder handles dev dependencies and compilation; runtime handles production deps and global CLI tools. Don't blur the line.
  • Never assume binaries exist. Just because something is installed during the build phase doesn't mean it's available at runtime. Explicitly install global CLIs in the runtime stage.
  • Lockfiles matter. npm ci is strict by design — if you use it, make sure package-lock.json is in the build context.
  • Debug top-to-bottom. Each error layer revealed the next. Fix the first thing, rebuild, repeat.
This journey turned a "secure by default" cloud-native container setup into a WSL2-friendly development environment — proving that with persistence and systematic debugging, even the most restrictive environments can be tamed. The 🦞 was well earned.
